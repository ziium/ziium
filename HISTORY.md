# History

이 문서는 지음의 버전별 변화와 단계 경계를 요약한다. `DECISIONS.md`가 설계 판단의 이유를 기록한다면, `HISTORY.md`는 각 버전에서 무엇이 확정되었고 다음 단계가 무엇을 맡는지 정리한다.

현재 저장소는 `v0.1` 코어 위에 `v0.2` 연구 원칙이 고정되었고, `v0.3`에서 코어 문법이 확장된 상태다.

## v0.1 - 문장형 코어와 실행 경로를 세운 단계

### 성격

- 현재 구현의 기준선이다.
- controlled language로서의 최소 실행 가능 범위를 확정한 단계다.
- 문서 초안만 있는 상태를 넘어, 실행기와 테스트가 함께 유지되는 상태를 목표로 했다.

### 핵심 변화

- Rust 기반 `lexer + normalizer + parser + resolver + hir lowering + interpreter + CLI/REPL` 파이프라인을 세웠다.
- 값 바인딩, 재대입, 출력, `쉬기`, `if/while`, 함수 정의와 괄호 호출, 목록/레코드/인덱싱, `의` 속성 접근을 실행 가능한 코어 문법으로 정리했다.
- structured diagnostic, 회귀 테스트, 샘플 프로그램, 브라우저 playground를 함께 유지하기 시작했다.
- 표면 문법은 한국어 문장형으로 두되, 내부 표현은 점차 `Send` 중심 의미 모델로 정리하기 시작했다.

### 현재 포함 범위

- 안정된 코어 문법
- 닫힌 built-in 메시지 집합
- 제한된 실험 프레임
  - `X으로 Y`
  - `X 더하기 Y`
  - `<대상>에 <값> 추가`
  - `<함수>를 <레코드>로 호출한다`
  - `...에서 맨위/맨뒤/맨앞 요소를 꺼낸 것이다`
- 브라우저 `그림판` host binding과 캔버스 하노이탑 데모

### 규격상 의미

- 지음은 한글 키워드 번역판이 아니라 한국어 문장 흐름을 가진 controlled language로 정의된다.
- 메시지 이름은 전역 예약어가 아니라, 좁은 문맥에서만 해석되는 built-in 집합으로 유지된다.
- truthiness, 자유 자연어 추론 parser, 열린 메시지 집합은 도입하지 않는다.

### 아직 남아 있는 제한

- 타입체커가 없다.
- 바이트코드 VM이나 LLVM/Cranelift 백엔드가 없다.
- 패키지/모듈 시스템이 없다.
- 실험 프레임은 아직 언어 전체 규격으로 일반화되지 않았다.

## v0.2 - 메시지 프레임 연구 경계를 고정한 단계

### 성격

- 새 기능을 대량으로 연 버전이라기보다, 후속 확장의 규칙과 금지선을 명확히 한 단계다.
- 사용자에게 보이는 표면 문법 변화보다, 해석 우선순위와 내부 의미 모델의 경계를 정리하는 데 무게를 둔다.

### 핵심 변화

- 서로 다른 실험 프레임을 내부적으로 `receiver + selector + arguments` 형태의 `Send` 공통형으로 설명하기로 확정했다.
- `A의 B`는 항상 속성 접근으로 먼저 읽고, built-in noun message 해석은 lowering/runtime 단계의 제한 규칙으로 미루기로 했다.
- `A으로 B`는 모든 호출 표기가 아니라 unary transform frame으로 제한하기로 했다.
- 상태 변경 프레임과 결과 서술 프레임은 조회/계산 프레임보다 더 좁은 닫힌 집합으로 관리하기로 했다.
- 메시지 이름은 계속 폐쇄형 built-in 집합으로 유지하기로 했다.
- `...꺼낸 것이다`와 별도로 statement-only effect frame `...꺼낸다`를 닫힌 형태로 분리했다.
- HIR의 `SendSelector`는 `Property/Transform`의 열린 표면 selector와 `Word/Keyword/Resultive`의 닫힌 built-in selector로 더 분명히 나뉘었다.
- 위치 명시형 `꺼내기`의 첫 목록 일반화로 `맨뒤 요소를 꺼낸 것이다/꺼낸다`, `맨앞 요소를 꺼낸 것이다/꺼낸다`를 추가했다.

### 규격상 의미

- 겉으로 보이는 문법 변화는 작지만, 언어가 문장을 읽는 우선순위와 확장 정책은 더 엄격하게 정리됐다.
- `의 / 으로 / 에 / 더하기` 같은 프레임은 같은 연구 축에서 다루되, 자유 자연어 호출 문법으로 일반화하지 않는다.
- 새 문법을 열 때는 먼저 어떤 `receiver`, `selector`, `arguments`로 낮출지 설명해야 한다.

### v0.2가 맡는 다음 작업

- `의` 프레임의 property-first parsing과 built-in noun fallback 규칙을 테스트와 구현 경계에 더 직접 반영
- `으로` 프레임의 unary transform 제한을 회귀 테스트로 고정
- 상태 변경 및 결과 서술 프레임의 닫힌 범위를 유지하면서 후속 확장 순서를 정리

### 주의

- `v0.2`는 아직 완성된 새 릴리스라기보다, 연구 방향과 규격 경계를 문서로 고정한 단계다.
- 따라서 `v0.2`의 의미는 "문법이 많이 늘었다"보다 "어떤 문법을 앞으로 어떻게 늘릴지 원칙이 정리됐다"에 가깝다.

## v0.3 - v0.2 연구 위에 코어 문법을 넓힌 단계

### 성격

- 메시지 프레임은 건드리지 않고, 제어 구문과 바인딩 표면 문법을 확장한 단계다.
- 코어 실행 경로(조건문, 반복문, 바인딩, 재대입)의 빈 영역을 채우는 데 집중했다.

### 핵심 변화

- 리스트 인덱스 대입: `목록[인덱스]를 값으로 바꾼다` 문법으로 목록 원소를 제자리 수정할 수 있게 되었다.
- 적용 바인딩: `결과는 5를 두배한 것이다` 형태로 함수 적용 결과를 바인딩할 수 있게 되었다.
- 단문 if-else: `x == 0이면 y를 돌려주고 아니면 z를 돌려준다`처럼 한 줄에 조건 분기를 쓸 수 있게 되었다.
- 가변/불변 바인딩: `이다`는 const(불변), `넣는다`는 let(가변)으로 구분되었다. 불변 바인딩에 재대입하면 에러가 발생한다.
- for-each 반복문: `목록의 각각 항목에 대해` 형태로 목록을 순회할 수 있게 되었다. while + 인덱스 패턴 없이 간결한 반복이 가능하다.
- 존재 바인딩: `바구니에 ["사과"]가 있다`, `바구니에는 ["사과"]가 있다` 형태로 컬렉션에 더 자연스러운 한국어 바인딩을 쓸 수 있게 되었다. `이다`와 동일한 const 의미다.

### 규격상 의미

- 새 키워드 4개 추가: `각각`, `대해`, `있다`, `넣는다`/`넣고`.
- 닫힌 메시지 집합(`SendSelector`)은 변경하지 않았다.
- `ForEach` AST/HIR variant가 추가되었으나, 메시지 시스템 바깥의 제어 구문이다.
