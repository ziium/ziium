# Vision

## 한 문장 비전

한국어 화자가 문장처럼 읽고, 기계는 엄격한 구조로 해석할 수 있는 프로그래밍 언어를 만든다.

## 우리가 만들고 싶은 것

이 언어는 "한글 키워드 세트"가 아니라 "한국어 문장형 코드"를 지향한다. 핵심은 표면이 자연스럽게 보이는 것만이 아니다. 조사, 서술어, 어순이 코드 의미를 분명히 드러내도록 설계해야 한다.

예를 들어 아래 둘은 비슷해 보이지만 역할이 다르다.

```txt
나이는 20이다
나이를 21로 바꾼다
```

첫 문장은 바인딩이고, 둘째 문장은 상태 변경이다. 이 차이를 조사와 서술어가 드러내야 한다.

## 핵심 철학

### 1. 문장 단위는 한국어답게

한 줄의 코드는 가능한 한 하나의 한국어 문장처럼 보여야 한다.

- 바인딩은 서술문처럼
- 조건문은 조건절처럼
- 함수 정의는 설명문처럼
- 오류 메시지는 교정문처럼

### 2. 내부 구조는 절대 느슨하지 않게

표면이 자연스럽다고 해서 의미 해석이 모호해지면 안 된다. v0.1에서는 다음을 명확히 제한한다.

- 주어 생략 자동 복원 금지
- 조사 없는 암시적 문장 해석 금지
- 문맥에 따라 완전히 다른 의미가 되는 문법 금지
- "자연어처럼 보이지만 파서가 임의 추론해야 하는 형식" 금지

### 3. 한국어다움은 장식이 아니라 의미 구조

`은/는`, `이/가`, `을/를`, `의`, `로/으로` 같은 조사는 구문적 역할을 드러낸다. 이 언어에서는 조사를 가능하면 문법 요소로 사용한다.

### 4. v0.1은 하이브리드 전략을 택한다

완전한 자연어형 수식은 구현 난도가 매우 높다. 따라서 v0.1에서는 문장 수준은 한국어형으로, 표현식 내부는 일부 기호 연산을 허용한다.

예:

```txt
합은 왼쪽 + 오른쪽이다
나이 > 19이면
  "성인"을 출력한다
```

이 선택은 철학의 후퇴가 아니라 구현 순서의 문제다.

### 5. 한국어 오류 경험도 언어의 일부

이 언어는 소스 코드만 한국어여서는 안 된다. 오류 메시지, 제안, 도움말, REPL 출력도 같은 경험을 유지해야 한다.

## 하지 않을 것

- 기존 언어의 모든 키워드를 한글로 번역한 뒤 끝내지 않는다.
- v0.1에서 자유로운 자연어 이해를 목표로 하지 않는다.
- 조사 생략과 문장 압축을 과도하게 허용하지 않는다.
- C-like 블록 문법을 겉만 한글로 포장하지 않는다.
- 구현 전에 미학만 논의하지 않는다. 명세, 테스트, 실행 모델을 함께 전진시킨다.

## 설계 판단 기준

문법 제안이나 구현 선택이 나왔을 때 아래 질문에 답해야 한다.

1. 이 문장은 한국어 화자가 읽기에 역할이 드러나는가?
2. 파서가 문맥 추측 없이 안정적으로 해석할 수 있는가?
3. 오류가 났을 때 한국어로 교정 가능한가?
4. AI가 명세를 보고 일관되게 코드를 생성할 수 있는가?
5. v0.1 범위 안에서 구현 가능한가?

## 성공한 모습

성공한 v0.1은 아래 상태를 만족한다.

- 낯선 사용자가 예제 10개를 읽고 7개 이상 동작을 예측할 수 있다.
- 파서와 인터프리터가 핵심 문장을 일관되게 처리한다.
- 오류 메시지가 한국어 문장형 코드와 어울린다.
- 새 문법을 추가할 때도 `LANGUAGE_SPEC.md`와 `GRAMMAR.ebnf`를 함께 갱신하는 문화가 정착된다.
