# Language

지음의 v0.1 언어 문서를 한곳에 모은다. 이 문서는 철학, 표면 문법, 토큰화 규칙, 실행 의미, 메시지 경계, 진단 원칙을 함께 설명한다. 파서 구현용 형식 문법은 `GRAMMAR.ebnf`를 기준으로 본다.

## 목표와 범위

- 지음은 "한글 키워드 번역판"이 아니라 한국어 문장 흐름을 중심에 둔 언어를 지향한다.
- 문장 단위는 한국어답게 보이되, 내부 표현은 엄격하게 유지한다.
- v0.1은 자연어 이해가 아니라 제한된 controlled language를 목표로 한다.
- 현재 구현 범위는 Rust 기반 `lexer + normalizer + parser + resolver + interpreter + CLI/REPL`이다.

## 핵심 원칙

- 블록은 들여쓰기로 표현한다.
- 조건식과 반복식은 오직 불리언만 허용한다.
- 바인딩과 재대입은 표면 문장이 달라야 한다.
- `의`는 표준 속성 접근 표기다.
- 함수 정의 표기는 `<이름> 함수는`을 유지한다.
- 문장 끝 `.`은 선택적이지만, 블록을 여는 줄에는 쓰지 않는다.

## 현재 문법 계층

지음의 현재 구현은 모든 문법이 같은 안정도를 갖지 않는다. 지금은 코어 문법과 실험 프레임을 구분해서 보는 편이 정확하다.

### 안정된 코어

- 바인딩과 재대입
- 출력과 쉬기
- 조건문과 반복문
- 함수 정의와 괄호 호출
- 목록/레코드/인덱싱
- `의`를 이용한 일반 속성 접근
- 기호 연산과 기본 비교/논리 연산

### 실험 프레임

- unary transform frame: `X으로 Y`
- binary word frame: `X 더하기 Y`
- keyword verb frame: `<대상>에 <값> <동작>`
- statement named call frame: `<함수>를 <레코드>로 호출한다`
- host keyword frame: `그림판에 <레코드>로/으로 <동작>`
- resultive bind frame: `...에서 맨위 요소를 꺼낸 것이다`
- resultive effect frame: `...에서 맨위 요소를 꺼낸다`

현재 구현은 이 실험 프레임들을 허용하지만, 자유 자연어 문법으로 넓히지 않는다. 모두 닫힌 built-in 집합과 제한된 위치에서만 쓴다.

## 어휘 규칙

### 공백과 주석

- 인코딩은 UTF-8 기준이다.
- 줄바꿈은 문장 경계에 관여한다.
- 들여쓰기는 공백으로만 처리한다. 탭은 허용하지 않는다.
- `#` 뒤의 내용은 줄 끝까지 주석이다.

### 식별자

- 한글 식별자를 기본으로 허용한다.
- 숫자로 시작하는 이름은 허용하지 않는다.
- 조사처럼 보이는 끝음절을 가진 이름도 가능하지만, 모호성을 줄이기 위해 긴 이름을 권장한다.

### 리터럴

- 숫자: `0`, `1`, `20`
- 문자열: `"안녕"`
- 불리언: `참`, `거짓`
- 목록: `[1, 2, 3]`
- 레코드: `{ 이름: "영희", 나이: 18 }`
- 레코드 shorthand: `{ 이름, 나이: 18 }`

레코드에서 식별자 이름만 적으면 같은 이름의 값을 가리키는 shorthand로 읽는다.

```txt
{ 이름, 나이: 18 }
```

는 아래와 같다.

```txt
{ 이름: 이름, 나이: 18 }
```

- shorthand는 식별자 키에서만 허용한다.
- 문자열 키는 항상 `키: 값` 형식을 써야 한다.

### 조사와 문장 종결

- `은/는`, `이/가`, `을/를`, `의`, `에`, `으로` 같은 조사는 문법 일부로 취급한다.
- lexer는 붙은 조사를 longest-match에 가깝게 분리하되, 한글 이름 모호성은 normalizer와 parser 문맥으로 보정한다.
- 완료된 문장 뒤에는 선택적으로 `.`를 붙일 수 있다.

## 문장

### 바인딩

```txt
이름은 "철수"이다.
숫자들은 [1, 2, 3]이다.
원반은 시작탑의 원반들에서 맨위 요소를 꺼낸 것이다.
```

- `...은/는/들은 ...이다` 형태를 쓴다.
- 같은 스코프에서 같은 이름을 다시 바인딩하면 오류다.
- 현재는 좁은 결과 서술 바인딩으로 `...에서 맨위 요소를 꺼낸 것이다`도 허용한다.

### 재대입

```txt
점수를 점수 + 1로 바꾼다.
```

- 이미 정의된 이름만 재대입할 수 있다.
- 재대입 문장은 바인딩과 다르게 `...를 ...로 바꾼다`를 사용한다.

### 출력

```txt
이름을 출력한다.
숫자들의 길이를 출력한다.
```

- 전치형 `출력 "하하"`는 지원하지 않는다.
- v0.1 기본형은 `표현식 + 을/를 + 출력한다`다.

### 결과 서술 effect statement

```txt
시작탑의 원반들에서 맨위 요소를 꺼낸다.
```

- 현재는 닫힌 effect frame으로 `...에서 맨위 요소를 꺼낸다`만 허용한다.
- 이 문장은 값을 버리고 상태만 바꾼다.
- 내부 의미는 `...에서 맨위 요소를 꺼낸 것이다`와 같은 `pop last` 계열이지만, 반환값은 바인딩하지 않는다.

### 쉬기

```txt
0.5초 쉬기.
1초 쉬기.
```

- `쉬기`는 문장이다.
- 현재 표면 문법은 `표현식 + 초 + 쉬기`다.
- 실행 의미는 "초"라는 이름 붙은 시간 인자를 받는 형태로 본다.
- CLI/REPL에서는 실제로 기다리고, 브라우저 데모에서는 실행 이벤트를 지연 재생한다.

### 반환

```txt
왼쪽 + 오른쪽을 돌려준다.
```

- `돌려준다`는 함수 본문 안에서만 쓸 수 있다.

### 조건문

```txt
나이 >= 20이면
  "성인"을 출력한다.
아니면
  "미성년자"를 출력한다.
```

- 조건식은 반드시 불리언이어야 한다.
- `아니면`은 선택적이다.

### 반복문

```txt
인덱스 < 숫자들의 길이인 동안
  숫자들[인덱스]을 출력한다.
  인덱스를 인덱스 + 1로 바꾼다.
```

- `...인 동안` 블록을 사용한다.
- 반복문 안에서 처음 바인딩한 이름은 반복문 밖에서 definite binding으로 보지 않는다.

### 함수 정의

```txt
합치기 함수는 왼쪽, 오른쪽을 받아
  왼쪽 + 오른쪽을 돌려준다.

시작 함수는 아무것도 받지 않아
  "시작합니다"를 출력한다.
```

- 함수 이름을 먼저 쓰고 `함수는`으로 이어진다.
- 매개변수가 없으면 `아무것도 받지 않아`를 쓴다.

### 키워드 메시지 문장

```txt
과일들에 "감" 추가.
그림판에 { 배경색: "#f6efe2" }으로 지우기.
그림판에 { x: 120, y: 80, 색: 빨강 }으로 점찍기.
```

- 현재 v0.1에서 키워드 메시지는 built-in으로 닫혀 있다.
- 현재 특별 취급하는 형태는 `<목록>에 <값> 추가`와 `그림판에 <레코드>로/으로 <동작>`이다.

### 이름 붙은 호출문

```txt
탑옮기기를 { 원반수: 원반수 빼기 1, 시작, 보조, 목표 }로 호출한다.
```

- 현재는 문장 자리에서만 허용한다.
- 오른쪽 인수는 레코드여야 한다.
- 현재는 사용자 함수에만 사용할 수 있다.
- 레코드의 키 이름으로 함수 매개변수 이름을 맞춘다.
- 같은 이름이면 shorthand를 쓸 수 있다. 즉 `{ 시작, 보조 }`는 `{ 시작: 시작, 보조: 보조 }`와 같다.

## 표현식

### 기본 표현식

- 이름
- 리터럴
- 괄호식
- 함수 호출
- 인덱싱
- `의` 속성 접근

### 함수 호출과 인덱싱

```txt
합치기(3, 5)
"지음"으로 인사만들기
숫자들[0]
```

- 일반 함수 호출은 괄호 기반을 지원한다.
- 현재 프로토타입은 experimental unary transform frame으로 `X으로 Y`를 지원한다.
- `X으로 Y`는 현재 `Y(X)`로 낮춘다.
- 이 프레임은 당분간 단일 입력을 받아 새 값을 만드는 변환/생성 계열 연구 대상으로만 본다.

### 속성 접근

```txt
사람의 이름
사용자의 주소의 도시
5의 제곱
```

- `의`는 좌결합 후위 연산이다.
- `목록[0]의 이름` 같은 결합도 지원한다.
- parser는 `A의 B`를 항상 일반 속성 접근으로 먼저 읽고, built-in noun message 해석은 lowering/runtime의 제한 규칙에서만 수행한다.

### 연산자

- 기호 연산: `+`, `-`, `*`, `/`, `>`, `>=`, `<`, `<=`, `==`, `!=`
- 논리 연산: `그리고`, `또는`
- 단항 부정: `아니다`
- 단어 연산 메시지: `더하기`, `빼기`, `곱하기`, `나누기`

예:

```txt
합은 7 더하기 8이다.
값은 1 + 2 * 3이다.
로그인가능은 관리자 또는 편집자 그리고 활성이다.
```

### 우선순위

- 호출, 인덱싱, 속성 접근이 가장 강하다.
- 그다음 단항 부정, 곱셈/나눗셈, 덧셈/뺄셈, 비교, `그리고`, `또는` 순으로 읽는다.
- `더하기/빼기/곱하기/나누기`는 같은 연산 계층의 단어형 built-in으로 취급한다.

## 메시지 경계

지음은 메시지 지향 확장을 실험하고 있지만, 현재 메시지 이름은 전역 예약어가 아니다. 아주 좁은 문맥에서만 built-in 메시지로 해석한다.

실험 프레임의 표면 문법이 서로 달라도, 내부 의미는 모두 `receiver + selector + arguments` 형태의 제한된 `Send` 모델로 정리한다.

### 현재 닫힌 built-in 메시지 집합

- unary noun message: `길이`
- unary noun message: `제곱`
- binary word message: `더하기`, `빼기`, `곱하기`, `나누기`
- keyword verb message: `추가`
- host keyword message: `지우기`, `점찍기`, `사각형채우기`, `글자쓰기`

### `길이`

- `길이(값)`는 내장 함수다.
- `A의 길이`는 먼저 레코드 키 조회를 시도한다.
- 레코드 키가 없고 수신자가 목록이나 문자열이면 built-in 명사 메시지로 해석한다.
- 그 외에는 속성 오류다.

### `제곱`

- `A의 제곱`은 현재 정수와 실수에서만 built-in 명사 메시지로 해석한다.
- 정수는 정수 결과를, 실수는 실수 결과를 돌려준다.
- 그 외에는 속성 오류다.

### `더하기/빼기/곱하기/나누기`

- infix 위치에서만 built-in word operator로 해석한다.
- 함수 이름이나 일반 식별자로도 계속 사용할 수 있다.

### `추가`

- `추가(목록, 값)`와 `<목록>에 <값> 추가`에서만 특별 취급한다.
- 그 외 위치에서는 일반 식별자다.
- 상태를 바꾸는 프레임이므로 당분간 built-in 닫힌 집합으로만 유지한다.
- 후속 연구 후보로 `넣기`가 있지만, 현재 구현은 아직 `추가`만 지원한다.

### `그림판`

- `그림판`은 host binding 객체다.
- 현재 지원하는 문법은 `그림판에 <레코드>로/으로 <동작>`이다.
- 현재 지원하는 동작은 `지우기`, `점찍기`, `사각형채우기`, `글자쓰기`다.
- `지우기`는 `배경색` 또는 `색` 필드를 가진 레코드를 받는다.
- `점찍기`는 `x`, `y`, `색` 필드를 가진 레코드를 받고, 선택적으로 `크기`도 받을 수 있다.
- `사각형채우기`는 `x`, `y`, `너비`, `높이`, `색` 필드를 가진 레코드를 받는다.
- `글자쓰기`는 `글`, `x`, `y`, `색`, `크기` 필드를 가진 레코드를 받는다.
- `그림판` 동작은 host binding 부수 효과 프레임이므로 자유 자연어 동사로 확장하지 않는다.

### 우선순위 요약

1. 기존 문장 구조를 먼저 해석한다.
2. `의`는 항상 속성 접근으로 파싱한다.
3. `길이`와 `제곱`의 built-in lowering은 실행 시점에 결정한다.
4. 단어 연산은 infix 위치에서만 특별 취급한다.
5. `추가`, `그림판` 동작, 결과 서술 동사는 정확한 닫힌 프레임에서만 특별 취급한다.

## 실험적 호출 프레임과 후속 연구

현재 v0.1의 일반 함수 호출 표기는 `합치기(3, 5)`처럼 괄호 기반이다. 다만 현재 프로토타입은 제한된 실험으로 `X으로 Y`와 `정수/실수의 제곱`을 지원한다. 지음의 다음 단계에서는 "함수 호출을 한국어답게 보이게 하는가"보다 "조사별 의미 프레임을 호출 표면에 드러내는가"를 더 중요한 기준으로 본다.

### 연구 방향

- `X의 Y`: 성질, 조회, 파생값을 나타내는 unary noun frame
- `X으로 Y`: 단일 입력을 받아 새 값을 만드는 unary transform frame
- `대상에 값 Y`: 대상을 바꾸는 keyword verb frame
- `X 더하기 Y`: 계산을 나타내는 binary word frame
- `<함수>를 <레코드>로 호출한다`: statement-only named call frame
- `대상에서 목적어를 ...한 것이다`: 동작 결과를 명사구처럼 받는 result-descriptive frame

예:

```txt
문장은 "지음"으로 인사만들기이다.
결과는 5의 제곱이다.
과일들에 "감" 추가.
합은 3 더하기 5이다.
탑옮기기를 { 원반수: 3, 시작: "A", 보조: "B", 목표: "C" }로 호출한다.
```

### 결과 서술 문법 연구

지음의 다음 연구 후보 중 하나는, 동작 결과를 보다 한국어다운 명사구로 받는 결과 서술 문법이다.

예:

```txt
원반은 시작탑의 원반들에서 맨위 요소를 꺼낸 것이다.
```

이 표현은 `빼내기이다`보다 자연스럽지만, 단순 호출 sugar보다 더 큰 문법 계층을 연다. 핵심 요소는 다음 셋이다.

- 관형형 동사: `꺼낸`, `만든`, `찾은`
- 목적어 명시: `맨위 요소를`
- 명사화된 결과: `것이다`

이 연구는 `X의 Y`, `X으로 Y`, `대상에 값 Y`와는 다른 축이다. 앞의 셋이 조사 프레임 중심이라면, 이 문법은 "끝난 동작의 결과를 명사처럼 받는다"는 서술 프레임에 가깝다.

#### 권장하는 최소 범위

처음부터 자유 자연어를 열지 않고, 아주 좁은 부분집합만 허용하는 편이 맞다.

- 형태는 `<이름>은 <수신자>에서 <목적어구>를 <결과동사> 것이다`로 제한
- `것이다`는 자유 명사구가 아니라 고정된 결과 표지로 취급
- `결과동사`는 built-in 닫힌 집합으로 시작
- `<목적어구>`는 우선 식이 아니라 제한된 명사구 또는 role label로 취급

예를 들면 v0.2 실험 후보는 아래처럼 잡을 수 있다.

```txt
원반은 시작탑의 원반들에서 맨위 요소를 꺼낸 것이다.
문장은 이름에서 인사말을 만든 것이다.
```

#### 최소 문법 제안

첫 실험은 아래처럼 더 좁게 잡는 편이 맞다.

```txt
<바인딩이름>은 <수신자>에서 <역할명>을 <결과동사> 것이다.
```

여기서 중요한 점은 "이게 자유로운 한국어 문장"이 아니라, 고정된 틀을 가진 제한 문법이라는 점이다.

1. 허용 위치
   - 결과를 값으로 받는 형태는 우선 바인딩 오른쪽 값 위치에서만 허용
   - 즉 `원반은 시작탑의 원반들에서 맨위 요소를 꺼낸 것이다.`는 허용 후보
   - `시작탑의 원반들에서 맨위 요소를 꺼낸 것이다.` 같은 독립 결과 식 문장은 금지
   - 대신 문장형 상태 변경은 별도의 닫힌 frame `시작탑의 원반들에서 맨위 요소를 꺼낸다.`로 분리한다
2. 수신자 범위
   - 우선은 단순 이름 또는 `의` 속성 체인만 허용
   - 예: `시작탑`, `게임의 현재탑`
   - 산술식, 호출식, 비교식은 처음엔 금지
3. 역할명 범위
   - 자유 명사구가 아니라 닫힌 role label 집합으로 시작
   - 예: `맨위 요소`
   - `빨간 작은 원반` 같은 자유 수식 명사구는 금지
4. 결과동사 범위
   - 자유 동사 활용을 열지 않고, 닫힌 built-in 집합으로 시작
   - 예: `꺼낸`, `만든`, `찾은`
   - parser는 이들을 일반 형용사/동사가 아니라 전용 결과동사 토큰처럼 취급하는 편이 안전하다
5. 결과 표지
   - `것이다` 전체를 고정된 결과 표지로 본다
   - 내부적으로는 `것`과 `이다`를 따로 열지 않고, 최소 실험에서는 한 덩어리 표식처럼 다루는 편이 낫다

이 최소 문법을 지음 표면에 맞춰 다시 쓰면 사실상 이런 구조다.

```txt
bind_stmt = IDENT, TOPIC, resultive_expr, PERIOD? ;
resultive_expr = receiver, FROM, role_label, OBJECT, RESULT_VERB, RESULT_MARKER ;
receiver = IDENT | property_chain ;
```

즉 핵심은 "결과 서술 문법을 전체 자연어 구문으로 보지 않고, 바인딩 오른쪽에만 들어가는 제한된 식 문법과 statement-only effect frame으로 나눠 본다"는 점이다.

현재 구현된 첫 부분집합은 아래 둘이다.

```txt
원반은 시작탑의 원반들에서 맨위 요소를 꺼낸 것이다.
시작탑의 원반들에서 맨위 요소를 꺼낸다.
```

- 바인딩 오른쪽에서는 `...꺼낸 것이다`를 허용한다.
- 문장 자리에서는 `...꺼낸다`를 허용한다.
- 현재 수신자는 목록이어야 한다.
- `맨위 요소`와 `꺼낸/꺼낸다`만 지원한다.
- 의미는 기존 `마지막꺼내기(시작탑)`와 같은 `pop last`다.
- 두 형태 모두 목록을 실제로 바꾼다.
- `...꺼낸 것이다`는 값을 돌려주고, `...꺼낸다`는 값을 버린다.

#### 첫 구현 후보

처음 구현 후보는 하노이탑에 필요한 한 패턴만 열어도 충분하다.

```txt
원반은 시작탑의 원반들에서 맨위 요소를 꺼낸 것이다.
```

이 한 줄을 기준으로 하면 첫 버전의 닫힌 집합은 아래처럼 잡을 수 있다.

- `FROM`: `에서`
- `role_label`: `맨위 요소`
- `RESULT_VERB`: `꺼낸`
- `RESULT_MARKER`: `것이다`

이렇게 시작하면 사실상 "목록/탑에서 맨위 항목을 꺼내 결과로 바인딩한다"는 단일 built-in frame만 연 셈이 된다.

#### 금지할 형태

처음 단계에서는 아래를 금지하는 편이 맞다.

```txt
원반은 시작탑에서 1번째 원반을 꺼낸 것이다.
원반은 시작탑에서 아주 작은 빨간 원반을 꺼낸 것이다.
원반은 시작탑 + 보조탑에서 맨위 요소를 꺼낸 것이다.
시작탑의 원반들에서 맨위 요소를 꺼낸 것이다.
```

즉 순서 표현, 자유 수식 명사구, 복합 수신자, 독립 문장은 모두 나중으로 미룬다.

#### 내부 의미 제안

표면 문법은 자연어에 가깝게 두되, 내부 의미는 여전히 제한된 메시지/호출로 낮추는 편이 좋다.

예:

```txt
원반은 시작탑의 원반들에서 맨위 요소를 꺼낸 것이다.
```

는 내부적으로는 대략 이런 의미를 갖는다.

```txt
원반은 시작탑의 원반들에서 { 대상: "맨위 요소" }로 빼내기이다.
```

또는 더 단순하게는 아래와 같은 전용 AST로 표현할 수 있다.

```txt
Resultive {
  receiver: 시작탑의 원반들,
  patient: "맨위 요소",
  verb: "꺼낸",
}
```

즉 표면은 서술형으로 두고, 의미는 `receiver + role/patient + selector` 구조로 유지한다.

#### 왜 바로 구현하지 않는가

- 관형형 활용은 한국어 형태 변화가 크다.
- `것`을 일반 명사로 열면 parser가 빠르게 자연어 해석 쪽으로 흔들린다.
- `목적어구`를 자유롭게 열면 ordinary noun phrase와 일반 식별자 경계가 흐려진다.
- `꺼낸 것이다` 같은 표현은 조회가 아니라 상태 변경을 포함한 식이므로, side effect가 있는 표현식 설계도 함께 필요하다.

#### 현재 추천

- v0.1과 v0.2 초반에는 기존 임시형 또는 더 제한된 `에서` 프레임으로 버틴다.
- 결과 서술 문법은 `바인딩 RHS 전용 + 단순 수신자 + 닫힌 role label + 닫힌 결과동사 + 고정된 결과 표지`로만 시작한다.
- 첫 실험 후보는 하노이탑의 `시작탑의 원반들에서 맨위 요소를 꺼낸 것이다` 같은 도메인 뚜렷한 형태가 적합하다.

### 컬렉션 조작 동사 연구

아래 내용은 현재 구현된 기능이 아니라, 후속 연구에서 `추가`와 결과 서술 프레임을 어디로 확장할지 정리한 초안이다.

#### 기본 방향

- mutation을 동반하는 삽입은 `넣기` 축으로 연구한다.
- mutation을 동반하는 추출은 `꺼내기` 축으로 연구한다.
- 비파괴 조회는 `가져오기` 축으로 남겨 둔다.

즉 `가져오다`는 `get/fetch/peek` 계열 후보로 보고, `꺼내다`는 `pop/take/extract` 계열 후보로 본다.

#### 위치 표현

- 목록/큐 계열은 `맨앞`, `맨뒤`를 우선 검토한다.
- 탑/스택 계열은 `맨위`, `맨아래`를 우선 검토한다.
- `첫`, `마지막`은 더 일반적인 대체 표현 후보지만, 초반 연구는 방향 의미가 더 분명한 `맨앞/맨뒤/맨위/맨아래`를 우선한다.
- 후속 `넣기/꺼내기` 연구에서는 위치를 항상 명시하는 편을 기본 원칙으로 둔다.

#### `넣기`

`넣기`는 초반 연구에서 위치를 항상 드러내는 형태로 여는 편이 맞다.

예:

```txt
목록의 맨앞에 1 넣기
목록의 맨뒤에 3 넣기
탑의 맨위에 원반 넣기
```

- 위치를 항상 적으면 기본 방향이라는 숨은 규칙을 들이지 않아도 된다.
- 이 축은 `push_front/push_back/insert` 같은 더 정밀한 조작 후보로 본다.
- 다만 현재 구현은 아직 `넣기`를 열지 않았고, 기존 built-in `추가`만 지원한다.

#### `꺼내기`

`꺼내기`도 초반 연구에서 위치를 항상 드러내는 형태를 기본으로 두는 편이 더 안전하다.

예:

```txt
값은 목록에서 맨앞 요소를 꺼낸 것이다.
값은 목록에서 맨뒤 요소를 꺼낸 것이다.
원반은 탑에서 맨위 요소를 꺼낸 것이다.

목록에서 맨앞 요소를 꺼낸다.
목록에서 맨뒤 요소를 꺼낸다.
탑에서 맨위 요소를 꺼낸다.
```

- 결과를 값으로 받는 식은 `...꺼낸 것이다` 축으로 본다.
- 결과를 버리고 상태만 바꾸는 문장은 `...꺼낸다` 같은 effect statement 축으로 분리해서 본다.
- 위치 없는 `꺼내기`는 기본 방향이라는 숨은 규칙을 요구하므로, 초반 연구에서는 열지 않는 편이 맞다.
- 현재 구현은 아직 일반 위치형 `꺼내기`를 열지 않았고, 닫힌 frame `...에서 맨위 요소를 꺼낸 것이다/꺼낸다`만 지원한다.

#### `가져오기`

`가져오기`는 당분간 mutation을 동반하는 추출에는 쓰지 않는 편이 맞다.

예:

```txt
이름은 사람에서 이름을 가져온 것이다.
값은 설정에서 포트를 가져온 것이다.
```

- 이 축은 `get/fetch/peek`처럼 값을 읽되 원본을 바꾸지 않는 의미에 더 잘 맞는다.
- 따라서 `꺼내기`와 `가져오기`를 같은 동사군으로 섞지 않는 편이 좋다.

### 정의 문법과 호출 문법의 관계

이 연구 방향에서도 함수 정의는 계속 현재 형식을 유지한다.

```txt
인사만들기 함수는 이름을 받아
  "안녕, " + 이름 + "!"을 돌려준다.
```

즉 정의는 "함수가 어떤 매개변수를 받는가"를 드러내고, 호출은 "사용자가 그 함수를 어떤 조사 프레임으로 읽는가"를 드러낸다. 두 문법이 반드시 같은 조사를 공유할 필요는 없다.

```txt
문장은 "지음"으로 인사만들기이다.
```

### 현재 판단

- `X의 Y`는 가장 유망하다.
- `X으로 Y`는 현재 구현에서는 unary call lowering으로 동작하지만, 장기적으로는 변환/생성 계열에 제한하는 편이 좋다.
- `대상에 값 Y`는 현재 `추가`처럼 변경 동작 계열에서 자연스럽다.
- `X 더하기 Y`는 당분간 닫힌 built-in 집합으로 유지하는 편이 안전하다.
- 상태 변경과 결과 서술 프레임은 조회/계산 프레임보다 더 좁게 관리하는 편이 맞다.
- 컬렉션 조작 연구는 장기적으로 `추가` 하나에 기대기보다 `넣기/꺼내기`와 위치 프레임으로 분리하는 편이 더 유망하다.
- 컬렉션 조작 연구는 기본 위치 sugar보다 위치 명시 원칙을 먼저 세우는 편이 낫다.
- `가져오기`는 mutation이 없는 조회 계열 후보로 남겨 두는 편이 좋다.

### 금지할 확장 방식

- 모든 함수 이름을 모든 조사 프레임에서 자동 허용
- 정의 문법을 호출 프레임에 맞춰 억지로 바꾸기
- 자연어 추론으로 생략된 조사나 숨은 목적어를 복원하기

이 연구는 `호출 문법의 전면 자연어화`가 아니라, `조사별 의미 프레임을 제한적으로 표면에 올리는 일`로 본다.

## 실행 의미

### 처리 파이프라인

```txt
source -> lexer -> normalizer -> parser -> resolver -> hir lowering -> interpreter
```

### 스코프와 이름 해석

- resolver가 미정의 이름, 같은 스코프 재정의, 함수 밖 반환을 먼저 잡는다.
- `이면/아니면`은 양쪽 분기 모두에서 바인딩된 이름만 블록 뒤에서 사용 가능하다.
- 함수 안의 자유 변수는 바깥 스코프 캡처를 허용한다.

### 값 의미

- 불리언은 truthiness 없이 `참/거짓`만 허용한다.
- 목록은 인덱싱, 길이 조회, `추가`를 지원한다.
- 목록은 `마지막꺼내기`도 지원한다.
- 레코드는 키 기반 `의` 접근을 지원한다.
- 함수는 클로저처럼 동작한다.

### 현재 내장 기능

- `길이(목록 또는 문자열)`
- `추가(목록, 값)`
- `마지막꺼내기(목록)`
- 목록/문자열의 `의 길이`
- 정수/실수의 `의 제곱`
- 목록에 대한 `에 ... 추가`
- `그림판`에 대한 `에 { ... }로/으로 ...`
- unary transform frame `X으로 Y`

## 진단 원칙

- 오류는 어휘/구문/이름 해석/실행 단계로 구분한다.
- 가능한 경우 줄/열 span을 포함한다.
- 실행 오류는 호출 경로를 함께 보여준다.
- 코드 프레임은 오류 줄과 주변 문맥을 함께 보여준다.

예:

```txt
[실행 오류]
위치: 2번째 줄 5번째 열
메시지: 조건식은 불리언이어야 합니다.
```
