# Message Boundaries

이 문서는 지음의 현재 프로토타입에서 메시지 이름과 일반 식별자의 경계를 어떻게 해석하는지 고정한다.
목표는 문법을 더 자연스럽게 넓히더라도 parser와 사용자가 같은 규칙을 보게 하는 것이다.

## 범위

현재 구현이 지원하는 메시지 계열은 아래뿐이다.

- unary noun message: `목록의 길이`, `문자열의 길이`
- binary word message: `더하기`, `빼기`, `곱하기`, `나누기`
- keyword verb message: `<목록>에 <값> 추가`

이 집합은 현재 `closed built-in set`으로 취급한다. 즉 이 문서에 적히지 않은 메시지 이름은 자동으로 메시지로 해석하지 않는다.

## 기본 원칙

메시지 이름은 "단어 그 자체"로 예약되지 않는다. `문맥`이 메시지 해석을 결정한다.

즉:

- `더하기 함수는 ...`에서 `더하기`는 함수 이름이다.
- `더하기(7, 8)`에서 `더하기`는 호출 가능한 이름이다.
- `7 더하기 8`에서만 `더하기`가 built-in binary word message가 된다.

같은 원칙이 `길이`, `추가`에도 적용된다.

## 현재 해석 규칙

### 1. `길이`

- `길이(값)`은 built-in 함수 호출이다.
- `A의 길이`는 unary noun message 후보다.
- 단, 실제 해석은 수신자 종류에 따라 달라진다.

현재 우선순위:

1. 수신자가 레코드면 항상 일반 키 조회를 먼저 시도한다.
2. 수신자가 목록 또는 문자열이면 built-in noun message `길이`를 허용한다.
3. 그 외 값은 속성 오류다.

따라서:

- `과일들의 길이`는 길이 계산
- `문장의 길이`는 길이 계산
- `사람의 길이`는 레코드 키 `"길이"`가 있을 때만 성공

### 2. `더하기`, `빼기`, `곱하기`, `나누기`

이 이름들은 `infix slot`에서만 built-in binary word message다.

예:

```txt
합은 7 더하기 8이다.
차는 10 빼기 3이다.
```

반대로 아래에서는 일반 식별자다.

```txt
더하기 함수는 왼쪽, 오른쪽을 받아
  왼쪽 + 오른쪽을 돌려준다

합은 더하기(7, 8)이다.
```

즉 "ordinary identifier를 전역 예약어로 뺏지 않고, infix 위치에서만 built-in으로 읽는다"가 현재 규칙이다.

### 3. `추가`

`추가`는 두 문맥에서만 특별 취급한다.

- `추가(목록, 값)` : built-in 함수 호출
- `<목록>에 <값> 추가` : built-in keyword verb message

그 외 위치에서는 일반 식별자다.

예:

```txt
추가(과일들, "감")
과일들에 "감" 추가
```

현재 keyword message는 `에 ... 추가` 한 형태만 허용한다. `에서`, `에게`, `로` 등을 쓰는 다른 동사 프레임은 아직 없다.

## 우선순위 규칙

현재 parser와 문서가 따라야 하는 우선순위는 아래와 같다.

1. 함수 정의, 바인딩, 재대입, 출력, 조건문, 반복문 같은 기존 문장 구조를 먼저 본다.
2. `의`는 표면적으로 항상 속성 접근으로 읽는다.
3. `A의 길이`의 내장 lowering 여부는 parser가 아니라 runtime dispatch가 결정한다.
4. `더하기/빼기/곱하기/나누기`는 infix 위치에서만 built-in word operator다.
5. `추가`는 `에 <표현식> 추가` 형태에서만 keyword message다.
6. 알 수 없는 메시지 이름을 일반 함수 호출로 암시 변환하지 않는다.

즉 "메시지처럼 보이는 단어를 최대한 많이 추측"하는 방향이 아니라, "정확한 문맥에서만 좁게 허용"하는 방향이다.

## 허용 예시

```txt
합은 7 더하기 8이다.
과일들에 "감" 추가.
과일들의 길이를 출력한다.
```

```txt
더하기 함수는 왼쪽, 오른쪽을 받아
  왼쪽 더하기 오른쪽을 돌려준다.

결과는 더하기(3, 5)이다.
```

## 금지 예시

```txt
값은 1이다.
값 추가 2
```

이유:

- `추가` keyword message는 현재 `에` 프레임만 허용한다.

```txt
합은 7 최대값 8이다.
```

이유:

- `최대값`은 built-in binary word message 집합에 없다.

```txt
사람은 { 이름: "영희" }이다.
사람의 길이
```

이유:

- 레코드에는 `"길이"` 키가 없고, 레코드에는 length lowering을 적용하지 않는다.

## 미래 확장 원칙

사용자 정의 메시지를 열기 전에는 아래를 먼저 정해야 한다.

- 메시지 선언 문법
- selector namespace
- built-in과 user-defined selector 충돌 규칙
- keyword verb frame 선언 방식
- `Send` 중심 AST/HIR 도입 여부

그 전까지는 built-in 집합을 닫아 두는 편이 안전하다.
