- `나이가 19보다 크면`은 v0.2 이상 후보

## 2026-03-16 - 함수 정의는 `<이름> 함수는` 형식을 채택

### 결정 내용

함수 정의 헤더는 `함수 더하기는 ...`가 아니라 `더하기 함수는 ...` 형식을 사용한다. 이때 조사는 `함수는`으로 고정한다.

### 이유

이 형식은 이름을 먼저 드러낸 뒤 그 이름의 종류가 함수임을 설명하는 한국어 명사구에 더 가깝다. 문장형 언어의 톤에도 더 잘 맞는다.

### 대안

- `함수 더하기는 ...`
- `더하기는 함수로서 ...`

### 영향

- 함수 헤더 EBNF 순서가 `IDENT FUNCTION FUNCTION_TOPIC`이 된다.
- 예제와 테스트 문구를 모두 같은 형식으로 유지해야 한다.

## 2026-03-16 - v0.2 문법 연구는 메시지 중심 축으로 진행

### 결정 내용

v0.2 이후 자연스러운 한국어 문법 확장은 ad hoc sugar를 늘리는 대신, message-oriented 모델을 중심으로 연구한다.

### 이유

`목록의 길이`, `목록에 값 추가`, `7 더하기 8` 같은 문법을 함수 호출, 속성 접근, 연산자의 별도 sugar로 각각 추가하면 문법이 조각나기 쉽다. 반대로 메시지 전송을 핵심 의미로 두면 셋을 같은 축에서 설명할 수 있다.

### 대안

- 함수 호출 sugar를 기능별로 따로 추가
- 기호 연산 중심 모델을 유지한 채 한국어 표면 문법만 덧붙이기

### 영향

- unary noun message, binary word message, keyword verb message를 별도 연구 대상으로 둔다.
- v0.2 문서에서는 `Send` 중심 AST와 dispatch 모델을 검토한다.
- 메시지 문법 연구와 현재 규칙은 `LANGUAGE.md`에 함께 반영한다.

## 2026-03-16 - 현재 메시지 집합은 폐쇄된 built-in 집합으로 해석

### 결정 내용

현재 프로토타입에서 메시지 이름은 전역 예약어로 취급하지 않고, 좁은 문맥에서만 built-in message로 해석한다. 또한 현재 메시지 집합은 `길이`, `제곱`, `더하기/빼기/곱하기/나누기`, `추가`, 그리고 `그림판` host 동작으로 닫아 둔다.

### 이유

메시지 이름을 넓게 열면 ordinary identifier와 빠르게 충돌하고, parser가 과도하게 추측하게 된다. 한국어 문장형 문법은 자연어처럼 보여도 controlled language로 유지해야 한다.

### 대안

- 메시지 이름을 전역 예약어처럼 고정
- 반대로 모든 식별자를 infix/verb 문맥에서 메시지로 열어 두기

### 영향

- `더하기`는 infix 위치에서만 built-in word operator다.
- `추가`는 `에 <표현식> 추가`와 `추가(...)`에서만 특별 취급한다.
- `그림판`은 `그림판에 <레코드>로/으로 <동작>`에서만 특별 취급한다.
- `A의 길이`는 record key lookup을 먼저 시도한 뒤, 목록/문자열/레코드에서만 built-in noun message로 해석한다.

## 2026-03-17 - 브라우저 캔버스는 `그림판` host binding으로 연다

지음이 브라우저에서 바로 동작할 때, Canvas API를 영문 이름 그대로 노출하지 않고 `그림판` host binding으로 감싼다. 현재 v0.1 범위에서는 `그림판에 <레코드>로/으로 <동작>`만 지원하고, 동작 집합은 `지우기`, `점찍기`, `사각형채우기`, `글자쓰기`로 제한한다.

이유:

브라우저 호스트 API를 그대로 노출하면 지음의 표면 문법이 깨진다. 반대로 `그림판에 { ... }으로 사각형채우기` 같은 형태는 기존 `대상에 값 동작` 프레임을 유지하면서도 브라우저와의 접점을 만들 수 있다. 또한 named parameter를 당장 별도 문법으로 열지 않고, 레코드 인자 하나로 제한하면 파서와 실행기의 복잡도를 낮출 수 있다.

채택:

- `그림판`은 내장 host 객체로 제공한다.
- 캔버스 명령 인수는 당분간 레코드 하나만 받는다.
- `로/으로`는 현재 그림판 메시지에서 명세 레코드와 동작 이름 사이 경계로 허용한다.
- 브라우저 데모의 캔버스 하노이탑은 지음 코드가 직접 `그림판`을 호출해서 프레임을 만든다.
- `점찍기`는 짧은 canvas 예제를 위한 기본 동작으로 함께 제공한다.
- `A의 제곱`은 정수/실수에서만 built-in noun message로 해석한다.
- 구체 규칙은 `LANGUAGE.md`의 메시지 경계 섹션을 따른다.

## 2026-03-17 - 레코드 리터럴은 동일 이름 shorthand를 허용

### 결정 내용

레코드 리터럴에서 식별자 이름만 적으면 같은 이름의 값을 가리키는 shorthand로 허용한다. 예를 들어 `{ 시작, 목표 }`는 `{ 시작: 시작, 목표: 목표 }`와 같다.

### 이유

지음의 다음 호출 연구에서 `{ 원반수: 원반수 빼기 1, 시작, 보조, 목표 }` 같은 이름 붙은 인수 표기가 자연스럽게 나오고 있다. 이때 같은 이름을 반복해서 적는 부담을 줄이려면 record shorthand가 먼저 있는 편이 더 일관적이다. 또한 새 기호를 들이지 않고 기존 레코드 문법 안에서 해결할 수 있다.

### 대안

- 항상 `이름: 값`만 허용
- shorthand를 호출 문법에만 한정

### 영향

- `{ 이름 }`는 `{ 이름: 이름 }`로 해석한다.
- shorthand는 식별자 키에서만 허용한다.
- 문자열 키는 계속 `\"키\": 값` 형식을 써야 한다.

## 2026-03-17 - 이름 붙은 호출은 statement-only frame으로 제한

### 결정 내용

`<함수>를 <레코드>로 호출한다` 형식을 실험적으로 지원하되, 현재는 문장 자리에서만 허용한다. 이 문법은 사용자 함수에만 적용하고, 레코드 키 이름으로 매개변수 이름을 맞춘다.

### 이유

`탑옮기기(원반수 빼기 1, 시작, 목표, 보조)` 같은 괄호 호출은 재귀 예제에서 읽기 부담이 크다. 반면 `<함수>를 <레코드>로 호출한다`는 이름 붙은 인수 역할을 드러내면서도, 이미 들어간 레코드 리터럴 및 shorthand 규칙과 잘 맞는다. 다만 이를 일반 표현식 호출까지 넓히면 값 자리 문법이 무거워지므로, 우선은 statement-only frame으로 제한하는 편이 안전하다.

### 대안

- 괄호 호출만 유지
- `<함수>를 값들로 호출한다` 같은 위치 인수형 문장 호출
- 일반 표현식 위치까지 named call을 즉시 개방

### 영향

- `탑옮기기를 { 원반수: 3, 시작: "A", 보조: "B", 목표: "C" }로 호출한다.`를 허용한다.
- 현재는 사용자 함수에만 허용하고, built-in 함수에는 적용하지 않는다.
- 레코드 키 누락이나 추가 키는 런타임 오류로 진단한다.

## 2026-03-17 - 후속 호출 문법 연구는 조사별 의미 프레임으로 진행

### 결정 내용

v0.2 이후 호출 문법 연구는 "괄호 호출을 한국어 단어로 치환"하는 방향보다, `의 / 으로 / 에 / 더하기` 같은 조사 및 단어 프레임을 호출 표면에 올리는 방향으로 진행한다. 이때 함수 정의 문법은 계속 `<이름> 함수는 ...을 받아` 형식을 유지한다.

### 이유

`"지음"으로 인사만들기`, `5의 제곱`, `과일들에 "감" 추가`, `3 더하기 5`는 서로 다른 의미 프레임을 자연스럽게 드러낸다. 반대로 모든 호출을 한 가지 한국어 문법으로 통일하려 하면 문법이 인공적으로 느껴지거나 빠르게 모호해진다. 또한 정의 문법까지 호출 프레임에 맞춰 바꾸면 함수 헤더가 불필요하게 복잡해진다.

### 대안

- 괄호 호출만 유지하고 조사 기반 호출은 연구하지 않음
- 모든 단일 인자 호출을 `X으로 Y`로 일반화
- 함수 정의도 `이름으로 받아`처럼 호출 프레임에 맞춰 바꿈

### 영향

- `X의 Y`는 unary noun frame 연구 대상으로 본다.
- `X으로 Y`는 unary transform frame 연구 대상으로 보되, 변환/생성 계열에 제한하는 쪽을 우선 검토한다.
- `대상에 값 Y`는 keyword verb frame 확장 축으로 본다.
- `X 더하기 Y`는 binary word frame으로 보되, 당분간 built-in 닫힌 집합을 유지한다.
- 함수 정의 예제는 계속 `인사만들기 함수는 이름을 받아`처럼 유지한다.

## 2026-03-17 - unary transform frame과 `제곱`을 제한적으로 실험

### 결정 내용

현재 프로토타입은 제한된 실험으로 `X으로 Y`와 `정수/실수의 제곱`을 지원한다. `X으로 Y`는 현재 unary callable에 대한 lowering으로 구현하고, `제곱`은 숫자 전용 built-in noun message로 해석한다.

### 이유

`"지음"으로 인사만들기`와 `5의 제곱`은 지음의 방향을 잘 드러내면서도 parser와 interpreter에 비교적 작은 비용으로 들어간다. 특히 `X으로 Y`는 함수 정의 문법을 바꾸지 않고 호출 문법만 더 한국어답게 실험할 수 있게 한다.

### 대안

- 괄호 호출만 유지
- `제곱`도 일반 함수 호출로만 두기
- `X으로 Y`를 아직 문서 연구 단계에만 남기기

### 영향

- unary callable은 `X으로 Y` 형태로도 호출할 수 있다.
- `Y` 자리는 현재 bare callee 이름만 허용한다.
- 이 표기는 일반 다인자 호출의 대체가 아니며, 추가 인수는 계속 괄호 호출이나 다른 frame으로 표현한다.
- 숫자는 `A의 제곱` 형태를 지원한다.
- 이 기능은 향후 transform 계열로 제한될 수 있으므로, 아직 실험적 표기로 본다.

## 2026-03-17 - `0.5초 쉬기`를 이름 붙은 시간 인자의 첫 사례로 도입

### 결정 내용

`쉬기`는 일반 함수 호출이 아니라 `표현식 + 초 + 쉬기` 형태의 독립 문장으로 도입한다. 현재 `초`는 시간 단위이자 이름 붙은 인자 표면의 첫 사례로 본다.

### 이유

`쉬기(0.5)`보다 `0.5초 쉬기`가 지음의 문장형 톤에 더 잘 맞는다. 동시에 `초`를 단순 장식이 아니라 이름 붙은 시간 인자로 해석하면, 나중에 named parameter 문법으로 일반화하기도 쉽다. 브라우저에서는 실제 blocking sleep 대신 실행 이벤트 지연으로 재생해야 하므로, 출력/프레임/대기시간을 같은 타임라인에 두는 편이 맞다.

### 대안

- `쉬기(0.5)`처럼 괄호 호출로만 도입
- 브라우저에서는 `쉬기`를 무시하고 CLI에서만 실제 sleep
- `밀리초`, `분` 같은 단위를 처음부터 함께 설계

### 영향

- parser는 `표현식 + 초 + 쉬기`를 정식 문장으로 해석한다.

## 2026-03-18 - 후속 `꺼내기` 일반화는 `맨뒤 요소`, 그 다음 `맨앞 요소`로 연다

### 결정 내용

현재 닫힌 frame `맨위 요소를 꺼낸 것이다/꺼낸다`의 후속 일반화는 `맨뒤 요소`, 그 다음 `맨앞 요소`까지 진행했다. 다음 후보는 `맨밑 요소`다.

### 이유

현재 구현의 `맨위 요소`는 내부적으로 `pop last`에 해당한다. 따라서 일반 컬렉션 쪽 첫 확장을 `맨뒤 요소`로 잡으면 현재 의미와 가장 가깝고, 새 표면 문법을 열더라도 내부 selector와 런타임 primitive를 가장 적게 흔들 수 있다. 그 다음 `맨앞 요소`를 열어야 목록 쪽 위치어가 앞/뒤 쌍으로 균형을 갖는다. `맨밑 요소`는 여전히 세로 은유가 강해서 일반 목록 문법과는 조금 떨어져 있다.

### 대안

- `맨밑 요소`를 `맨뒤 요소`보다 먼저 여는 안
- `맨위/맨앞/맨뒤/맨밑`을 한 번에 모두 개방
- `맨위 요소`를 일반 목록의 기본 위치어처럼 계속 사용

### 영향

- 현재 구현은 `값은 목록에서 맨뒤 요소를 꺼낸 것이다`, `목록에서 맨뒤 요소를 꺼낸다`, `값은 목록에서 맨앞 요소를 꺼낸 것이다`, `목록에서 맨앞 요소를 꺼낸다`를 지원한다.
- 다음 구현 후보는 `값은 탑에서 맨밑 요소를 꺼낸 것이다`, `탑에서 맨밑 요소를 꺼낸다`다.
- `맨밑 요소`는 코어 위치어 집합에는 남기되, 첫 일반화 단계에서는 열지 않는다.
- `맨위 요소`는 현재처럼 탑/스택형 수신자와 닫힌 기존 frame에서 유지한다.

## 2026-04-15 - 리스트 인덱스 대입: IndexAssign variant 채택

### 결정 내용

`목록[인덱스]를 값으로 바꾼다` 문법을 `Stmt::IndexAssign` 별도 variant로 구현한다. 기존 `Stmt::Assign`의 target을 `AssignTarget` enum으로 일반화하는 대안을 채택하지 않는다.

### 이유

1. 기존 `Assign { name: String, value }` 핸들링 코드를 건드리지 않아 blast radius가 최소다.
2. 코드베이스의 flat enum 스타일(`Bind`, `Assign`, `Print`, `Return` 등 동격 variant)과 일치한다.
3. 향후 Property 대입이 필요하면 동일 패턴으로 variant를 추가할 수 있다.

### 대안

- `Assign`의 `name: String`을 `target: AssignTarget`으로 일반화 — 한 번에 모든 대입 형태를 통합하지만, 기존 코드 전면 수정 필요
- 파서에서 `IndexAssign`을 `Assign`으로 desugar (base 변수 전체를 재대입) — 리스트의 Rc<RefCell> 참조 의미론과 충돌

### 영향

- 파서의 `Object` 분기에서 `Expr::Index` 후 `Direction + Change` 패턴을 인식하는 분기 추가
- 기존 `parse_named_call_statement`를 인라인하여 IndexAssign/NamedCall을 같은 위치에서 분기
- 중첩 인덱스(`목록[i][j]`), 레코드 필드 대입, 음수 인덱스 래핑은 명시적으로 defer
- 상대적 변화(`만큼 줄인다/늘린다`)도 인덱스 재대입에 적용

## 2026-04-15 - 적용 바인딩: `X를 Y한 것이다` 파서 접근 채택

### 결정 내용

### 이유

- 노멀라이저에서 `Ident("두배한")` → `Ident("두배") + Applied("한")` 분리 — 새 토큰 종류 필요, `한`으로 끝나는 합법적 식별자에 거짓 양성 가능
