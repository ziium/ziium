## 2026-03-16 - 언어 이름을 `지음(ziium)`으로 확정

### 결정 내용

언어의 공식 이름은 `지음`, 로마자 표기는 `ziium`으로 사용한다.

### 이유

문서와 구현에서 더 이상 가칭 상태를 유지할 이유가 없고, 사용자 노출 명칭과 구현 패키지 이름을 일치시키는 편이 협업과 배포 모두에서 명확하다.

### 대안

- 가칭 상태를 더 유지
- 한글 이름만 쓰고 로마자 표기를 두지 않음

### 영향

- README와 명세 문서의 표기를 `지음(ziium)`으로 통일한다.
- Cargo 패키지, 라이브러리, CLI 바이너리 이름을 `ziium`으로 사용한다.
- 기본 파일 확장자는 `.zm`을 사용한다.

## 2026-03-16 - 문장 수준 한국어, 표현식 수준 하이브리드

### 결정 내용

v0.1은 문장 단위 문법을 한국어형으로 두고, 표현식 내부는 일부 기호 연산을 허용한다.

### 이유

완전한 자연어형 수식은 구현 난도가 높고, parser 명세가 쉽게 흔들린다. 먼저 안정적인 실행 가능 범위를 확보하는 것이 우선이다.

### 대안

- 완전 자연어형 수식까지 한 번에 설계
- 반대로 문장도 기호 중심으로 단순화

### 영향

- `나이 > 19이면`은 v0.1 허용
- `나이가 19보다 크면`은 v0.2 이상 후보

## 2026-03-16 - `의`를 표준 속성 접근 연산자로 채택

### 결정 내용

`의`는 v0.1부터 정식 후위 연산자로 지원한다.

### 이유

`의`는 한국어 문장형 코드의 정체성을 가장 잘 드러내는 요소 중 하나이며, 레코드 접근에도 자연스럽다.

### 대안

- 점 표기 `.` 사용
- 문자열 키 인덱싱만 허용

### 영향

- lexer가 붙은 조사 분리를 지원해야 한다.
- parser는 좌결합 후위 연산을 구현해야 한다.

## 2026-03-16 - 조건식 truthiness 금지

### 결정 내용

조건문과 반복문은 오직 불리언 값만 허용한다.

### 이유

문장형 언어일수록 암시적 규칙이 많아지기 쉽다. truthiness를 허용하면 자연스러운 문장처럼 보여도 실제 규칙이 숨는다.

### 대안

- Python 유사 truthiness 채택

### 영향

- `개수이면`은 오류
- `개수 > 0이면`처럼 명시적 비교를 요구

## 2026-03-16 - 블록은 들여쓰기 기반

### 결정 내용

중괄호 대신 들여쓰기로 블록을 표현한다.

### 이유

문장 흐름을 더 자연스럽게 유지할 수 있고, `이면`, `아니면`, `인 동안`과 잘 어울린다.

### 대안

- `{}` 블록
- `끝` 토큰

### 영향

- lexer가 INDENT/DEDENT를 생성해야 한다.
- 탭 금지 규칙이 필요하다.

## 2026-03-16 - 함수 정의는 `<이름> 함수는` 형식을 채택

### 결정 내용

함수 정의 헤더는 `함수 더하기는 ...`가 아니라 `더하기 함수는 ...` 형식을 사용한다. 이때 조사는 `함수는`으로 고정한다.

### 이유

이 형식은 이름을 먼저 드러낸 뒤 그 이름의 종류가 함수임을 설명하는 한국어 명사구에 더 가깝다. 문장형 언어의 톤에도 더 잘 맞는다.

### 대안

- `함수 더하기는 ...`
- `더하기는 함수로서 ...`

### 영향

- 함수 헤더 EBNF 순서가 `IDENT FUNCTION FUNCTION_TOPIC`이 된다.
- 예제와 테스트 문구를 모두 같은 형식으로 유지해야 한다.

## 2026-03-16 - v0.2 문법 연구는 메시지 중심 축으로 진행

### 결정 내용

v0.2 이후 자연스러운 한국어 문법 확장은 ad hoc sugar를 늘리는 대신, message-oriented 모델을 중심으로 연구한다.

### 이유

`목록의 길이`, `목록에 값 추가`, `7 더하기 8` 같은 문법을 함수 호출, 속성 접근, 연산자의 별도 sugar로 각각 추가하면 문법이 조각나기 쉽다. 반대로 메시지 전송을 핵심 의미로 두면 셋을 같은 축에서 설명할 수 있다.

### 대안

- 함수 호출 sugar를 기능별로 따로 추가
- 기호 연산 중심 모델을 유지한 채 한국어 표면 문법만 덧붙이기

### 영향

- unary noun message, binary word message, keyword verb message를 별도 연구 대상으로 둔다.
- v0.2 문서에서는 `Send` 중심 AST와 dispatch 모델을 검토한다.
- 메시지 문법 연구와 현재 규칙은 `LANGUAGE.md`에 함께 반영한다.

## 2026-03-16 - 현재 메시지 집합은 폐쇄된 built-in 집합으로 해석

### 결정 내용

현재 프로토타입에서 메시지 이름은 전역 예약어로 취급하지 않고, 좁은 문맥에서만 built-in message로 해석한다. 또한 현재 메시지 집합은 `길이`, `더하기/빼기/곱하기/나누기`, `추가`로 닫아 둔다.

### 이유

메시지 이름을 넓게 열면 ordinary identifier와 빠르게 충돌하고, parser가 과도하게 추측하게 된다. 한국어 문장형 문법은 자연어처럼 보여도 controlled language로 유지해야 한다.

### 대안

- 메시지 이름을 전역 예약어처럼 고정
- 반대로 모든 식별자를 infix/verb 문맥에서 메시지로 열어 두기

### 영향

- `더하기`는 infix 위치에서만 built-in word operator다.
- `추가`는 `에 <표현식> 추가`와 `추가(...)`에서만 특별 취급한다.
- `A의 길이`는 record key lookup을 먼저 시도한 뒤, 목록/문자열에서만 built-in noun message로 해석한다.
- 구체 규칙은 `LANGUAGE.md`의 메시지 경계 섹션을 따른다.
