# Semantics

이 문서는 표면 문법이 내부 표현으로 어떻게 변환되고 실행되는지 설명한다. `LANGUAGE_SPEC.md`가 "무엇을 쓸 수 있는가"를 정의한다면, 이 문서는 "그 문장이 내부적으로 무엇이 되는가"를 정의한다.

## 1. 처리 파이프라인

v0.1 기준 권장 파이프라인:

1. Source text
2. Lexer
3. Parser
4. AST
5. Name resolution
6. Interpreter IR 또는 직접 실행
7. Runtime

중간 표현을 세분화한다면 AST 다음에 HIR을 둘 수 있다.

## 2. AST 핵심 노드

### 2.1 문장 노드

- `BindStmt(name, value)`
- `AssignStmt(name, value)`
- `PrintStmt(value)`
- `IfStmt(condition, then_block, else_block?)`
- `WhileStmt(condition, body)`
- `FunctionDefStmt(name, params, body)`
- `ReturnStmt(value)`
- `ExprStmt(expr)` for standalone call statements

### 2.2 표현식 노드

- `NameExpr(name)`
- `LiteralExpr(value)`
- `UnaryExpr(op, expr)`
- `BinaryExpr(left, op, right)`
- `CallExpr(callee, args)`
- `IndexExpr(base, index)`
- `PropertyExpr(base, name)`
- `ListExpr(items)`
- `RecordExpr(entries)`

## 3. 표면 문법에서 AST로

### 3.1 바인딩

표면:

```txt
이름은 "철수"이다
```

AST:

```txt
BindStmt(
  name="이름",
  value=LiteralExpr("철수"),
)
```

### 3.2 재대입

표면:

```txt
개수를 개수 + 1로 바꾼다
```

AST:

```txt
AssignStmt(
  name="개수",
  value=BinaryExpr(NameExpr("개수"), "+", LiteralExpr(1)),
)
```

### 3.3 조건문

표면:

```txt
나이 > 19이면
  "성인"을 출력한다
아니면
  "미성년자"를 출력한다
```

AST:

```txt
IfStmt(
  condition=BinaryExpr(NameExpr("나이"), ">", LiteralExpr(19)),
  then_block=[PrintStmt(LiteralExpr("성인"))],
  else_block=[PrintStmt(LiteralExpr("미성년자"))],
)
```

### 3.4 속성 접근

표면:

```txt
사람의 이름
```

AST:

```txt
PropertyExpr(
  base=NameExpr("사람"),
  name="이름",
)
```

중첩 속성 접근은 좌결합으로 묶는다.

### 3.5 독립 호출 문장

표면:

```txt
추가(숫자들, 3)
```

AST:

```txt
ExprStmt(
  expr=CallExpr(
    callee=NameExpr("추가"),
    args=[NameExpr("숫자들"), LiteralExpr(3)],
  ),
)
```

## 4. 이름 해석

### 4.1 스코프 규칙

- 전역 스코프는 파일 단위다.
- 함수 정의는 새 렉시컬 스코프를 만든다.
- 함수 매개변수는 함수 스코프에 바인딩된다.
- 조건문과 반복문은 v0.1에서 새 스코프를 만들지 않는다.
- 조건문에서 이름이 이후에도 유효하려면 모든 분기에서 바인딩되어야 한다.
- 반복문 본문에서 처음 바인딩된 이름은 반복문 뒤에서 자동으로 정의된 것으로 간주하지 않는다.

### 4.2 바인딩 조회

- 현재 스코프에서 찾는다.
- 없으면 바깥 스코프를 따라 올라간다.
- 끝까지 없으면 이름 오류다.

### 4.3 중복 바인딩

- 같은 스코프에서 같은 이름을 다시 `...은 ...이다`로 선언하면 오류다.
- 다른 내부 함수 스코프에서 같은 이름을 새로 정의하는 것은 허용한다.

### 4.4 definite binding

- 현재 줄에서 직접 실행되는 코드는 그 시점에 확실히 바인딩된 이름만 읽을 수 있다.
- 함수 본문은 외부 스코프의 나중 바인딩을 캡처할 수 있다.
- `이면 ... 아니면 ...`의 모든 분기에서 같은 이름을 바인딩하면, 그 이름은 조건문 뒤에서 사용 가능하다.
- `이면` 한쪽 분기에서만 바인딩한 이름은 조건문 뒤에서 사용할 수 없다.
- `인 동안` 본문에서만 바인딩한 이름은 반복문 뒤에서 사용할 수 없다.

## 5. 값 의미

### 5.1 불리언

조건문과 반복문의 조건은 반드시 불리언이어야 한다. 아래는 허용하지 않는다.

```txt
개수이면
  ...
```

`개수`가 숫자여도 조건식으로 자동 변환하지 않는다.

### 5.2 목록

- 순서가 있는 가변 시퀀스다.
- `추가(목록, 값)`은 목록을 제자리 수정한다.

### 5.3 레코드

- 문자열 키 기반 사전형 값이다.
- `사람의 이름`은 런타임에서 `"이름"` 키 조회와 대응된다.
- 키가 없으면 속성 접근 오류다.

## 6. 함수 의미

### 6.1 호출 모델

- 함수는 호출 시 새 프레임을 만든다.
- 인수는 좌에서 우로 평가한다.
- 매개변수 개수가 맞지 않으면 호출 오류다.

### 6.2 반환

- `돌려준다`를 만나면 현재 함수는 즉시 종료한다.
- 함수 본문 끝까지 `돌려준다`가 없으면 `없음`을 반환한다.

### 6.3 클로저

v0.1에서 익명 함수는 필수 아님이지만, 이름 있는 함수 정의는 외부 스코프를 읽을 수 있어야 한다. 쓰기 캡처는 구현 난도가 높다면 v0.1에서 제한할 수 있다.

## 7. `의` 연산의 의미

`의`는 단순한 한국어 소유격 표기가 아니라 표준 속성 접근 연산자다.

### 7.1 기본 규칙

`A의 B`는 내부적으로 `PropertyExpr(A, "B")`가 된다.

### 7.2 연쇄 규칙

`A의 B의 C`는 `PropertyExpr(PropertyExpr(A, "B"), "C")`다.

### 7.3 내장 속성

현재 구현은 일부 속성을 내장 동작으로 낮춘다.

예:

- `목록의 길이` -> `길이(목록)`
- `문자열의 길이` -> `길이(문자열)`

레코드는 이 lowering 대상에 넣지 않는다. 즉 레코드에서는 항상 일반 키 조회가 우선한다. `사람의 길이`는 레코드 키 `"길이"`가 있을 때만 성공한다.

### 7.4 키워드 메시지 lowering

현재 구현은 목록에 한해 아래 키워드 메시지를 지원한다.

```txt
과일들에 "감" 추가
```

의미는 아래와 같다.

```txt
추가(과일들, "감")
```

즉 표면 문법은 message-oriented 형태를 보이지만, 현재 런타임에서는 built-in 동작으로 바로 실행한다.

## 8. 평가 순서

표현식 평가 순서는 좌에서 우로 고정한다.

예:

```txt
f 함수는 값을 받아
  값을 출력한다
  값을 돌려준다

더하기(f(1), f(2))
```

위 코드는 `1`, `2` 순서로 출력되어야 한다.

## 9. 오류 의미

최소한 아래 오류 범주를 구분한다.

- LexError
- ParseError
- NameError
- TypeLikeError
- PropertyError
- CallError
- RuntimeError

v0.1이 정적 타입 시스템을 제공하지 않더라도, 실행 시 기대와 다른 값 종류를 만났을 때는 "타입 성격의 오류"로 분류할 수 있다.

## 10. 구현 권장 사항

### 10.1 AST와 문법 토큰 분리

조사 자체를 AST에 남길 필요는 없다. 파서 단계에서 이미 역할이 해석되었다면 AST는 의미 중심 노드로 단순화하는 편이 낫다.

### 10.2 진단용 정보 유지

각 AST 노드에는 최소한 소스 범위를 붙이는 것을 권장한다.

- 시작 줄/열
- 끝 줄/열
- 원본 토큰 범위

이 정보는 한국어 오류 메시지 품질에 직접 영향을 준다.

### 10.3 표면 문법과 내부 표현 분리

`...은 ...이다`와 `...를 ...로 바꾼다`는 표면적으로 비슷한 한국어 문장이지만 내부에서는 반드시 다른 노드가 되어야 한다. parser 단계에서 둘을 구분한 뒤 인터프리터가 다시 추정하지 않도록 한다.
