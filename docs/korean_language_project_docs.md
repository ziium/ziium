# 한국어 문장형 프로그래밍 언어 프로젝트 문서 세트

## 결론

이 프로젝트에서는 `PRD.md`가 **있으면 좋지만, 단독으로는 부족**합니다.

당신이 만들려는 것은 단순 앱 기능이 아니라:

- 새로운 프로그래밍 언어
- 새로운 문법 철학
- 파서/타입체커/인터프리터/컴파일러를 포함한 시스템
- Claude, Codex 같은 AI 에이전트와 협업하는 장기 프로젝트

이기 때문에, 실제로 더 중요한 것은 아래 네 가지입니다.

1. **비전과 원칙을 고정하는 문서**
2. **문법과 의미를 고정하는 언어 명세 문서**
3. **아키텍처와 구현 범위를 나누는 문서**
4. **AI가 바로 작업할 수 있는 작업 단위 문서**

즉, `PRD.md`는 있어도 되지만, 이 프로젝트에서는 ``**와 **``**가 더 핵심**입니다.

## 2. `PRD.md`

### 목적

제품/프로젝트 차원에서 “무엇을 왜 만드는가”를 정리한다.

### 필요한가?

**있으면 좋다.** 특히 Claude/Codex가 방향을 잃지 않게 하는 데 유용하다. 다만 앱 PRD처럼 기능 나열식으로 쓰면 안 되고, **언어 제품 PRD**로 써야 한다.

### 이 프로젝트에서 PRD가 답해야 할 질문

- 이 언어는 누구를 위한 것인가?
- 어떤 경험을 제공하려는가?
- 기존 프로그래밍 언어 대비 어떤 차별점이 있는가?
- 성공 기준은 무엇인가?
- v0.1에서 어디까지 구현할 것인가?
- 무엇은 아직 하지 않을 것인가?

### 권장 섹션

- 문제 정의
- 대상 사용자
- 핵심 사용자 경험
- 제품 목표
- 비목표
- v0.1 범위
- 성공 지표
- 위험 요소

### 비고

이 문서는 철학, 범위, 우선순위를 고정하는 문서다.

---

## 3. `VISION.md`

### 목적

언어 철학과 미학적 방향을 짧고 강하게 정리한다.

### 왜 중요한가

이 프로젝트는 기술 프로젝트이면서 동시에 **언어 설계 프로젝트**다. AI는 이런 종류의 프로젝트에서 자꾸 실용성만 최적화하려고 하므로, 철학 문서가 없으면 다시 C-like 문법으로 회귀하기 쉽다.

### 반드시 들어갈 것

- 이 언어가 추구하는 한국어다움
- 왜 조사와 문장형 문법이 중요한지
- 무엇을 “자연스럽다”고 볼 것인지
- 어떤 타협은 허용하고 어떤 타협은 거부할 것인지

### 핵심 문장 예시

- 이 언어는 “한글 키워드 번역”이 아니라 “한국어 문장형 코드”를 지향한다.
- 조사와 서술형 표현은 장식이 아니라 의미 구조의 일부다.
- 표면 문법은 자연스럽게 보이되, 내부 표현은 엄격해야 한다.

## 4. `LANGUAGE_SPEC.md`

### 목적

언어의 문법과 의미를 공식적으로 정의한다.

### 가장 중요함

이 프로젝트에서 가장 중요한 문서다. PRD보다도 중요하다. Claude/Codex가 코드 생성, 파서 수정, 테스트 작성, 예제 확장할 때 모두 이 문서를 기준으로 삼아야 한다.

### 반드시 들어갈 것

- 어휘 규칙
- 토큰 규칙
- 들여쓰기 규칙
- 조사 규칙
- 표현식 규칙
- 문장 규칙
- 선언/재대입 규칙
- 조건문 규칙
- 반복문 규칙
- 함수 정의 규칙
- 속성 접근(`의`) 규칙
- 오류 규칙
- 모호한 표현 금지 규칙

### 예시 섹션

1. 범위
2. 용어 정의
3. 어휘 규칙
4. 들여쓰기와 블록
5. 값과 변수
6. 표현식
7. 조건식
8. 반복문
9. 함수
10. 구조체/배열/사전
11. 내장 동작
12. 금지되는 표현
13. 파싱 우선순위
14. 예제

### 특히 중요한 하위 문서화 주제

- `의`는 단순 소유격이 아니라 접근 연산자
- `철수의 나이가 19보다 크고 철수가 학생이면` 같은 문장을 어떻게 분석하는지
- 조사 생략을 어디까지 허용하는지

## 5. `SEMANTICS.md`

### 목적

표면 문법이 내부적으로 어떤 의미 구조로 변환되는지 설명한다.

### 왜 필요한가

문장형 언어는 겉모습과 내부 구조가 다르다. AI가 파서만 보고 의미를 추정하면 자주 흔들린다. 따라서 아래를 명시해야 한다.

### 들어갈 것

- AST 개요
- HIR/MIR 개요
- 표면 문법 → 내부 표현 변환 규칙
- 이름 해석
- 스코프
- 타입 추론 규칙
- 내장 함수/내장 서술어의 lowering 규칙

### 예시

- `목록의 길이` → `len(목록)`
- `목록에 값을 추가` → `push(목록, 값)`
- `이름은 "철수"` → `bind(name="이름", value="철수")`

### 비고

이 문서는 파서 구현자, 타입체커 구현자, 백엔드 작업자 모두에게 중요하다.

## 10. `TASKS.md`

### 목적

AI에게 바로 줄 수 있는 작업 목록.

### 아주 중요

이 문서가 있어야 Claude/Codex가 “다음으로 뭘 할지”를 스스로 안정적으로 고를 수 있다.

### 한 작업 항목의 형식 예시

- 제목
- 배경
- 해야 할 일
- 건드릴 파일
- 완료 조건
- 제외 범위

### 예시

#### 작업: `의` 접근식 파서 추가

- 배경: `목록의 길이`, `철수의 나이`를 처리해야 한다.
- 해야 할 일: parser에 left-associative property access rule 추가
- 완료 조건:
  - `목록의 길이` 파싱 성공
  - `철수의 주소의 도시` 파싱 성공
  - AST snapshot 테스트 추가
- 제외 범위: 타입체커 수정은 하지 않음

## 11. `DECISIONS.md`

### 목적

주요 설계 결정을 기록한다.

### 왜 필요한가

언어 프로젝트는 같은 논의가 반복된다. AI도 이전 결정을 자꾸 뒤집는다.

### 형식

- 날짜
- 결정 제목
- 결정 내용
- 이유
- 대안
- 영향

### 예시 항목

- `의`는 property access 연산자로 채택
- 조건식의 주어 생략 복원은 v0.1에서 허용하지 않음
- 함수 정의 기본 문법은 `X 함수는 ...를 받아`

## 12. `TEST_PLAN.md`

### 목적

무엇을 어떻게 테스트할지 정의한다.

### 들어갈 것

- lexer 테스트
- parser golden test
- AST snapshot test
- interpreter 실행 테스트
- type error diagnostic test
- 문법 회귀 테스트

### 특히 중요한 것

문장형 언어는 회귀가 매우 쉽다. 자연스러운 표현 하나 추가하다가 다른 표현이 깨지기 쉽다. 그래서 snapshot 기반의 parser 테스트가 중요하다.

## 13. `ERRORS.md`

### 목적

오류 메시지 철학과 형식을 정의한다.

### 왜 중요한가

이 언어의 핵심은 “한국어다운 경험”이다. 오류 메시지가 영어식이거나 건조하면 전체 경험이 깨진다.

### 들어갈 것

- 오류 문체 원칙
- 파서 오류 예시
- 타입 오류 예시
- 제안 메시지 형식
- 위치 표시 형식

### 예시 원칙

- 무엇이 잘못되었는지 먼저 말한다.
- 기대한 문장 형태를 예시로 보여준다.
- 가능하면 수정 제안을 준다.

- `PRD.md`는 방향과 범위
- `LANGUAGE_SPEC.md`는 진짜 규칙
- `ARCHITECTURE.md`는 구현 구조

이렇게 역할을 나누는 게 가장 좋습니다.

````
# Vision

## 한 문장 비전

이 언어는 한글 키워드를 번역해 붙인 언어가 아니라, 한국어 문장처럼 읽히는 프로그래밍 언어를 지향한다.

## 핵심 철학

1. 한국어다움은 어휘가 아니라 문장 구조에서 온다.
2. 조사와 서술형 표현은 장식이 아니라 의미 구조의 일부다.
3. 표면 문법은 자연스럽게 보이되, 내부 의미 구조는 엄격해야 한다.
4. 자연어 전체를 허용하지 않고, 제한된 문장형 문법을 채택한다.
5. C-like 문법의 단순 번역으로 회귀하지 않는다.

## 우리가 만들고 싶은 경험

- 한국어 화자가 코드를 읽을 때 번역투가 아니라 문장처럼 느끼는 경험
- 초보자도 문장을 읽듯 이해할 수 있는 경험
- 오류 메시지까지 한국어다운 경험

## 하지 않을 것

- 완전 자유로운 자연어 프로그래밍
- 문장 해석을 추측에 의존하는 파서
- 영어권 언어의 문법을 얇게 번역한 표면적 한글화
````

````md
# Product Requirements Document

## 문제 정의

기존의 한글 기반 프로그래밍 언어 시도는 대부분 키워드만 한글로 바꾸고, 문장 구조는 영어권 언어 문법을 유지한다.
그 결과 한국어 화자에게는 여전히 번역투로 느껴지고, 조사와 서술형 표현이 사라져 어색하다.

## 제품 목표

이 프로젝트의 목표는 조사와 문장형 구문을 사용해 한국어처럼 읽히는 제한형 프로그래밍 언어를 만드는 것이다.

## 대상 사용자

- 한국어 화자 프로그래밍 학습자
- 언어 설계와 PL 연구에 관심 있는 개발자
- 한국어 기반 DSL/교육 언어를 실험하려는 개발자
- AI 에이전트와 함께 언어를 설계/구현하려는 개발자

## 핵심 사용자 경험

사용자는 코드를 읽을 때 다음처럼 느껴야 한다.
- 문장처럼 읽힌다.
- 각 문장 성분의 역할이 조사로 드러난다.
- 조건문과 반복문이 자연스럽다.
- 오류 메시지도 한국어답다.

## 대표 예시

```txt
이름은 "철수"
이름을 출력
목록에 값을 추가
나이가 19보다 크면
    "성인"을 출력
````

````
# Language Specification

## 1. 범위

이 문서는 지음(ziium)의 표면 문법과 의미를 정의한다.

## 2. 핵심 원칙

- 한 문장은 한 줄에 하나를 기본으로 한다.
- 블록은 들여쓰기로 표현한다.
- 조사 사용은 제한적으로 허용되며, 문법적 역할을 가진다.
- 표면 문법은 자연스럽지만, 내부 의미 구조는 엄격하다.

## 3. 어휘 규칙

### 3.1 식별자
- 한글, 영문, 숫자, 밑줄을 포함할 수 있다.

### 3.2 리터럴
- 문자열: `"철수"`
- 정수: `10`
- 실수: `3.14`
- 불: `참`, `거짓`
- 없음: `없음`

## 4. 블록

들여쓰기는 문장 블록을 형성한다.

예:
```txt
나이가 19보다 크면
    "성인"을 출력
````

## 5. 선언

형식:

```txt
이름은 "철수"
```

의미:

- `X은 Y` / `X는 Y`는 현재 범위에 새 이름 `X`를 바인딩한다.

## 6. 재대입

형식:

```txt
이름을 "영희"로 바꿈
```

## 7. 출력

형식:

```txt
이름을 출력
"안녕"을 출력
```

## 8. 속성 접근

형식:

```txt
목록의 길이
철수의 나이
철수의 주소의 도시
```

규칙:

- `의`는 속성/관계 접근 연산자다.
- `A의 B`는 왼쪽 결합한다.

## 9. 조건문

형식:

```txt
나이가 19보다 크면
    "성인"을 출력
아니면
    "미성년자"를 출력
```

## 10. 논리 연결

형식:

```txt
철수의 나이가 19보다 크고 철수가 학생이면
    "성인 학생"을 출력
```

규칙:

- `그리고/고`, `또는`은 조건절을 연결한다.
- v0.1에서는 생략된 주어 복원을 허용하지 않는다.

## 11. 함수 정의

기본 형식:

```txt
더하기 함수는 a와 b를 받아
    a + b를 반환
```

축약형은 추후 문법 설탕으로 추가할 수 있다.

## 12. 금지되는 표현

- 의미가 불명확한 주어 생략
- 조사 없는 문장을 기본 문법으로 사용하는 것
- 명세에 정의되지 않은 자유로운 자연어 표현

````

# Architecture

## 목표

이 문서는 언어 구현 구조를 설명한다.

## 전체 파이프라인

1. Source text
2. Lexer
3. Parser
4. AST
5. Name resolution
6. Type checking
7. HIR/MIR lowering
8. Interpreter and/or codegen
9. Diagnostics

## 모듈 구성 예시

```txt
/src
  /lexer
  /parser
  /ast
  /hir
  /types
  /mir
  /interp
  /codegen
  /diagnostics
  /repl
````

## 핵심 원칙

- 표면 문법과 내부 표현을 분리한다.
- 조사 기반 문장을 직접 AST primitive로 두지 않고 의미 단위로 낮춘다.
- 진단 메시지는 문장형 사용자 경험을 해치지 않도록 설계한다.

## 예시 lowering

- `목록의 길이` -> `len(목록)`
- `목록에 값을 추가` -> `push(목록, 값)`
- `이름은 "철수"` -> `bind(이름, "철수")`


```md
## 2026-03-16 — `의`를 접근 연산자로 채택

### 결정 내용
`의`는 속성/관계 접근 연산자로 취급한다.

### 이유
- `목록의 길이`, `철수의 나이` 같은 표현이 한국어 화자에게 자연스럽다.
- 내부적으로 field/property access로 lowering 가능하다.

### 대안
- 점(`.`) 기반 접근 유지
- 내장 함수 호출만 허용

### 영향
- parser에서 left-associative access 규칙이 필요하다.
- semantics 문서에서 lowering 규칙을 명시해야 한다.
```
