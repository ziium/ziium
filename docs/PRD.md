# Product Requirements Document

## 문제 정의

한국어 화자는 프로그래밍을 배울 때 두 겹의 장벽을 넘는다.

- 프로그래밍 개념 자체의 추상성
- 영어 기반 문법이 주는 언어적 거리감

기존의 한글 키워드 언어는 두 번째 문제를 부분적으로 줄였지만, 대체로 "영어 문법을 유지한 채 단어만 번역"하는 방식에 머물렀다. 이 프로젝트는 한국어 화자가 읽는 문장 흐름을 코드 표면 문법에 반영해, 읽기 경험 자체를 바꾸는 것을 목표로 한다.

## 제품 목표

- 한국어 문장형 코드의 최소 실용 문법을 설계한다.
- 구현 가능한 v0.1 범위를 정의하고 실제 실행기까지 이어질 문서를 만든다.
- 사람과 AI가 같은 기준으로 구현할 수 있도록 공식 명세와 작업 문서를 제공한다.

## 대상 사용자

- 한국어 기반 프로그래밍 교육을 실험하려는 개발자
- 프로그래밍 언어 설계 자체에 관심 있는 연구자와 제작자
- 한국어 문서 중심으로 코드를 읽고 쓰고 싶은 개인 사용자
- AI와 협업하며 새로운 언어를 빠르게 프로토타이핑하려는 팀

## 핵심 사용자 경험

사용자는 코드를 읽을 때 최소한 문장 단위에서는 "이 문장이 무엇을 한다"를 즉시 파악할 수 있어야 한다.

핵심 경험은 다음과 같다.

- 선언은 설명문처럼 읽힌다.
- 조건문은 조건절처럼 읽힌다.
- 함수 정의는 입력과 동작이 문장으로 드러난다.
- 오류 메시지는 문제와 수정 방향을 한국어로 제시한다.

## 대표 사용자 시나리오

### 시나리오 1: 간단한 규칙 계산

사용자는 나이, 점수, 상태 플래그를 바인딩하고 간단한 조건문으로 분기한다.

### 시나리오 2: 목록 처리

사용자는 목록을 만들고 반복문으로 순회해 값을 출력하거나 누적한다.

### 시나리오 3: 작은 도메인 스크립트 작성

사용자는 함수와 레코드를 이용해 작은 규칙 기반 스크립트를 만든다.

## 성공 기준

- 공식 문서만 읽고 parser 구현을 시작할 수 있다.
- 핵심 예제 프로그램이 인터프리터에서 실행된다.
- 문법 회귀 테스트가 최소 30개 이상 존재한다.
- 오류 메시지 예제가 실제 구현과 큰 차이 없이 유지된다.
- 새 기여자가 하루 안에 프로젝트 구조와 우선순위를 이해할 수 있다.

## 주요 위험

### 1. 자연스러움과 엄격성의 충돌

문장을 자연스럽게 만들수록 파서 규칙이 불안정해질 수 있다.

### 2. 범위 팽창

조사 처리, 어미 변화, 생략 복원까지 한 번에 하려 하면 v0.1이 무너진다.

### 3. 문서와 구현의 분리

문서가 앞서가고 구현이 따르지 못하면 설계가 쉽게 공중에 뜬다.

### 4. AI 협업 불안정

명세가 기계 친화적이지 않으면 AI가 비슷한 작업도 매번 다른 방식으로 구현한다.

## 완화 전략

- v0.1에서는 제한된 문장형만 허용한다.
- 명세 변경 시 `LANGUAGE_SPEC.md`와 `GRAMMAR.ebnf`를 함께 수정한다.
- 문법 추가 전에 parser golden test를 먼저 작성한다.
- `DECISIONS.md`에 설계 이유를 남겨 같은 논의를 반복하지 않는다.
