# Test Plan

이 문서는 v0.1 구현을 검증하는 테스트 전략을 정리한다. 문장형 언어는 작은 문법 변화가 넓은 회귀를 만들기 쉬우므로, 초기에 테스트 체계를 고정하는 것이 중요하다.

## 목표

- 문법 회귀를 빠르게 잡는다.
- 표면 문법과 내부 AST 사이의 불일치를 줄인다.
- 오류 메시지도 언어 기능처럼 테스트한다.

구체적인 lexer 예시는 `TOKENS.md`, parser 입력 세트는 `PARSER_TEST_CASES.md`를 기준으로 확장한다.

## 테스트 종류

### 1. Lexer 테스트

검증 대상:

- 식별자 인식
- 예약어 인식
- 붙은 조사 분리
- 문자열/숫자 리터럴
- 들여쓰기 토큰

대표 케이스:

- `사람의 이름`
- `나이를 20으로 바꾼다`
- `시작 함수는 아무것도 받지 않아`

### 2. Parser snapshot 테스트

검증 대상:

- 문장 구조
- 후위 속성 접근
- 함수 헤더
- 조건문 중첩
- 반복문 블록

대표 케이스:

- `사람의 주소의 도시`
- `나이 > 19이면 ... 아니면 ...`
- `더하기 함수는 왼쪽, 오른쪽을 받아`

### 3. Interpreter 실행 테스트

검증 대상:

- 바인딩
- 재대입
- 함수 호출
- 반복문
- 레코드 속성 접근

대표 케이스:

- 누적 합 계산
- 레코드 필드 출력
- 목록 길이와 인덱싱

### 4. 진단 테스트

검증 대상:

- 잘못된 들여쓰기
- 미정의 이름
- 잘못된 조건식
- 없는 속성 접근
- 인수 개수 불일치

### 5. 문서 예제 테스트

`README.md`, `SYNTAX_GUIDE.md`의 코드 블록과 `samples/*.zm`을 자동 검증하는 것이 이상적이다.

## 최소 테스트 수 목표

- lexer: 20
- parser snapshot: 30
- interpreter: 20
- diagnostics: 15

## 우선순위

1. parser snapshot
2. diagnostics
3. interpreter
4. 문서 예제 동기화

## 주의할 회귀 지점

- `의` 연쇄 접근이 인덱싱과 섞일 때
- `이면`과 `인 동안`의 경계
- `은/는`이 식별자 일부로 잘못 붙는 경우
- `아무것도 받지 않아` 함수 헤더 파싱

## 권장 테스트 포맷

- lexer: 입력 -> 토큰 나열
- parser: 입력 -> AST snapshot
- interpreter: 입력 -> 출력/반환값
- diagnostics: 입력 -> 오류 코드 + 메시지 + span
